# Inline Functions

## Overview
Kotlin's `inline` modifier is primarily an optimization for higher-order functions. The official docs explain that passing lambdas normally creates function objects and virtual calls, which can introduce overhead. Inlining copies the function body and inlinable lambdas into the call site to reduce that cost.

## Why inline exists
Higher-order functions are powerful, but they can allocate function objects and closures. Inlining can help in performance-sensitive code, especially for small higher-order utilities that are called frequently.

Example:

```kotlin
inline fun <T> lock(lock: Lock, body: () -> T): T {
  lock.lock()
  return try {
    body()
  } finally {
    lock.unlock()
  }
}
```

## When inline helps
- small higher-order helper functions
- collection-style utilities
- APIs that wrap setup and cleanup around a lambda

## Related keywords
- `noinline`: prevents a lambda parameter from being inlined
- `crossinline`: prevents non-local returns from that lambda

These are useful when some lambda parameters need different behavior.

## Tradeoff
Inlining reduces runtime overhead, but it can increase generated bytecode size. The Kotlin docs recommend using it reasonably rather than marking every function `inline`.

## Best practices
- Use `inline` mainly for small higher-order functions.
- Avoid using it automatically on large functions.
- Learn the difference between `inline`, `noinline`, and `crossinline`.
- Measure performance if optimization is the reason.

## Common mistakes
- Marking large ordinary functions as `inline` without benefit.
- Forgetting that code size can grow.
- Using `inline` without understanding lambda behavior.
